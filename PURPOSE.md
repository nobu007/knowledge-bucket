# PURPOSE — 到達目標と次の一手

## 北極星
あらゆる知識入力（Web記事、論文、Gitリポジトリ、メモ、PDF等）を不変ID付きのMarkdown正本（Git履歴管理）として安定配置し、再生成可能なインデックス・グラフキャッシュ（SQLite等）を活用することで、将来の人間とAIがいつでも検索・参照・再構成できる「永続知識基盤」を構築する。

## この文書の役割
この文書は Git Knowledge Bucket システムの現在の実装目標、完了条件、および次の具体的なアクション（優先順位付き）を定義する。20系/21系のコード改善エージェントはこの文書を入力として自律的に優先タスクを選択する。

## 目指す完成状態
1. **堅牢なデータストア**: 全ドキュメントがULID不変IDを持ち、`records/doc/{shard[0:2]}/{shard[2:4]}/{id}.md` へとSHA256ハッシュシャーディングされた物理配置で正しく保存されること。
2. **高速・正確な検索**: SQLite (FTS) 等をキャッシュインデックスとして用い、ファイルシステムを汚さず高速に全文・概念・近傍グラフ検索ができること。
3. **安全・クリーンなGit管理**: キャッシュや巨大バイナリをコミットせず、自動でバッチコミットし安全に変更を追跡できるGit同期パイプライン（`kb sync`）が機能していること。
4. **知的な概念接続 (Graph)**: AIが抽出した少数の主要・候補概念をもとに、出現頻度 (df/idf) や表記揺れ解決 (aliases)・Hub除外 (stop_concepts) を経て、文書間リンク・共起関係グラフがSQLite上で動的に再生成されること。
5. **優れたユーザーインターフェース**: 快適なCLI操作に加えて、検索・仮想ビュー・詳細表示・関係可視化を行えるローカルWeb UI（Next.js / Vite等）が完備されていること。

## 直近の優先成果

  - 言語（Node.js/TypeScript 推奨）、パッケージ構成、リンター・フォーマッター（ESLint, Prettier）、およびテスト環境（Vitest/Jest等）のセットアップ。
  - 基本ディレクトリ（`config/`, `prompts/`, `records/`, `scripts/`）と `.gitignore` の配置。

  - `kb init` (環境構築・DB初期化)、`kb add` (`.kb/inbox/` への一時保存) の実装。
  - `kb ingest` (入力タイプ判定、本文抽出、ID/ハッシュパス生成、Markdown正本出力、SQLiteインデックス登録) の実装。
  - `kb search` (SQLite FTSを用いた全文検索) の実装。
- **P3: バッチGit運用 & `kb sync` の実装**
  - ローカル変更のテスト、インデックスの差分同期、および Conventional Commits に準拠したバッチコミットを行う `kb sync` 機能の構築。

## 技術方針
1. **言語・環境**:
   - 保守性と堅牢性を確保するため、TypeScript (Node.js) を主要技術スタックとする。
   - 外部ライブラリは最小限にし、標準モジュールと軽量ライブラリを活用する。
2. **データベース/キャッシュ**:
   - `sqlite3` または `better-sqlite3` を使用して `.kb/index.sqlite` を構築する。
   - 全文検索には SQLite FTS5 拡張を活用する。
3. **ID・配置処理**:
   - ID生成には `ulid` パッケージを使用する。
   - `sha256(id)` を生成し、先頭4文字を2分割したディレクトリ構造を決定論的に決定する。
4. **エラーハンドリングとテスト**:
   - 各コンポーネント（特にID生成、パス解決、DB書き込み）に対して単体テストを徹底する。

## 完了の定義
1. CLIコマンド（`kb init`, `kb add`, `kb ingest`, `kb search`, `kb sync`）がローカル環境でエラーなく実行できること。
2. インポートされた各種ドキュメント（Web, メモ等）が `records/doc/` 配下のハッシュシャーディングされたディレクトリに正しい形式（ULID.md、Front Matter付き）で作成されること。
3. 作成されたドキュメントのメタデータおよび本文が `.kb/index.sqlite` にインデックス化され、`kb search` で高速に検索できること。
4. すべてのユニットテストがパスし、カバレッジが主要ロジック（パス計算、DBトランザクション）において 90% 以上であること。
5. `.kb/` 配下のキャッシュや一時ファイルが一切 Git 差分に含まれていないこと。
