# SYSTEM_CONSTITUTION.md

# Git Knowledge Bucket System Constitution

## 0. この文書の位置づけ

この文書は、Git Knowledge Bucket の最上位憲法である。

ここに書かれた内容は、実装・UI・CLI・インデックス方式・AIモデル・保存形式の細部が変わっても維持されるべき不変の設計原則である。

このシステムの目的は、情報を一時的に保存することではない。

このシステムの目的は、Web記事、GitHubリポジトリ、論文、PDF、メモ、設計資料、実験ログなどの知識資産を、将来の人間とAIが再利用できる形で、長期的・構造的・再生成可能にGitへ記録することである。

---

## 1. 究極のゴール

Git Knowledge Bucket の究極のゴールは、あらゆる外部情報・内部メモ・調査結果・技術知識・思考断片を、あとから人間とAIが検索・参照・再構成・接続・再利用できる「永続知識オブジェクト」として蓄積し続けることである。

ただし、それは人間向けのフォルダ整理システムではない。

このシステムは、Gitを「見やすい棚」として使うのではなく、「長期保存可能な正本ストレージ」として使う。

最終的に目指す状態は、次である。

```text
あらゆる知識入力
↓
不変IDを持つMarkdown正本
↓
Gitによる履歴管理
↓
再生成可能な検索インデックス
↓
再生成可能な概念グラフ
↓
人間とAIによる再利用
```

このシステムの価値は、単に情報を保存することではない。

価値の本体は、保存された情報が後から再び使えること、別の情報と接続できること、AIが文脈として取り出せること、人間が検証可能な正本へ戻れることである。

---

## 2. 一文定義

Git Knowledge Bucket は、あらゆる知識情報を不変ID付きMarkdown正本としてGitに保存し、検索・分類・概念グラフ・ベクトル索引などの派生構造をすべて再生成可能なキャッシュとして扱うことで、人間とAIが長期的に再利用できる知識基盤を構築するシステムである。

---

## 3. このシステムが解決する問題

人間の知識管理は、時間が経つと破綻する。

主な原因は次である。

- フォルダ分類が増えすぎる
- 1つの情報が複数カテゴリにまたがる
- あとから分類基準が変わる
- ファイル名やパスに意味を持たせすぎる
- タグを増やしすぎてノイズ化する
- AIに大量のラベル付けを任せて品質が崩れる
- インデックスやグラフを正本のように扱ってしまう
- 元情報と要約・解釈・関連情報の境界が曖昧になる
- 保存した情報が、後からAIに安全に渡せる形になっていない

Git Knowledge Bucket は、これらの問題を避けるために、情報の正本・識別子・物理配置・検索・分類・グラフ・AI分析の責務を明確に分離する。

---

## 4. 最重要思想

## 4.1 Gitは棚ではなく正本台帳である

Gitは、人間が目で見てフォルダを巡回するための棚ではない。

Gitは、知識オブジェクトの正本を保存し、変更履歴を保持し、復元可能性を担保する台帳である。

したがって、Git上の物理パスに意味を持たせてはならない。

物理パスは、人間の分類ではなく、永続IDから決定される安定配置でなければならない。

---

## 4.2 Markdownが正本である

各知識オブジェクトの正本は、1つのMarkdownドキュメントである。

Markdownは、人間が読める形式であり、Git差分を取りやすく、AIにも渡しやすく、長期保存に向いている。

検索インデックス、ベクトルDB、グラフDB、Parquet、SQLite、DuckDBなどは正本ではない。

それらは、Markdown正本から再生成可能なキャッシュである。

---

## 4.3 IDは不変である

すべての知識オブジェクトは、不変IDを持つ。

IDは、そのオブジェクトを長期的に参照するための唯一の安定識別子である。

本文が更新されても、タイトルが変わっても、カテゴリが変わっても、タグが変わっても、IDは変えてはならない。

IDが変わると、過去の参照・引用・グラフ・外部リンク・AIメモリとの対応が壊れる。

---

## 4.4 パスは意味ではなく配置である

ファイルパスは分類を表現しない。

ファイルパスは、IDから機械的に決定される物理配置である。

人間向けの意味分類は、フォルダではなく仮想ビューで表現する。

カテゴリ、タグ、関連概念、出典種別、著者、プロジェクト、関心領域、鮮度、重要度などは、インデックスやビューとして再構成する。

---

## 4.5 インデックスはキャッシュである

検索インデックス、FTS、ベクトルインデックス、グラフDB、Parquetエクスポート、ランキング結果は、すべて再生成可能でなければならない。

これらを正本にしてはならない。

インデックスが壊れても、Gitに保存されたMarkdown正本から復元できる状態を維持する。

---

## 4.6 グラフはMarkdownに直書きしない

知識間の関連は重要である。

しかし、関連リンクをMarkdown本文に大量に直書きしてはならない。

理由は、グラフ構造が変わるたびに大量のMarkdownファイルが書き換わり、Git差分が汚染されるからである。

Markdownに保存するのは、正本として必要な最小限のメタ情報と本文である。

関連構造は、プログラムがMarkdown正本と補助メタデータから再生成する。

---

## 4.7 AIは候補を出すが、正本構造を支配しない

AIは、要約、候補概念、関連候補、分類候補、重要論点、引用候補、注意点を生成してよい。

しかし、AIに構造全体を直接支配させてはならない。

AIの役割は、知識構造の候補を提示することである。

最終的なグラフ生成、インデックス生成、重複判定、正規化、禁止語処理、概念統合、ビュー構築は、決定論的なプログラムで制御する。

AIが出すタグや概念は少数でよい。

大量のタグをAIに生成させてはならない。

---

## 4.8 分類は後から変わる前提で設計する

知識の分類基準は必ず変わる。

今のカテゴリは未来のカテゴリではない。

したがって、物理構造をカテゴリに依存させてはならない。

カテゴリは後から何度でも作り直せる仮想ビューである。

同じ情報は、技術別、プロジェクト別、課題別、人物別、時系列別、重要度別、利用シーン別など、複数の切り口で再表示できなければならない。

---

## 5. 正本とキャッシュの境界

## 5.1 正本

正本とは、失われると復元できない情報である。

Git Knowledge Bucket における正本は、原則として次である。

- Markdownドキュメント
- 不変ID
- 出典情報
- 取得日時
- タイトル
- 本文または本文相当の抽出情報
- 人間またはAIが付与した最小限の解釈情報
- 正本に含めるべきメタデータ
- 必要な場合の引用・抜粋・要約
- その情報を保存した理由
- 将来再利用するための最小限の文脈

---

## 5.2 キャッシュ

キャッシュとは、正本から再生成可能な情報である。

次のものは正本ではない。

- SQLite index
- DuckDB database
- FTS index
- vector index
- graph database
- generated category views
- generated concept relations
- ranking results
- search result cache
- embedding cache
- temporary raw files
- crawler cache
- downloaded PDFs
- HTML snapshots
- log files
- temporary analysis results

キャッシュは壊れてもよい。

ただし、壊れたら再生成できなければならない。

---

## 6. データモデルの基本原則

## 6.1 1情報 = 1 Markdownドキュメント

各知識オブジェクトは、原則として1つのMarkdownドキュメントとして保存する。

1つのMarkdownは、1つの保存単位である。

保存単位は、あとからAIが文脈として取り出しやすく、人間が読んで意味を確認できる粒度でなければならない。

---

## 6.2 各ドキュメントは不変IDを持つ

各ドキュメントは、不変IDを持つ。

IDの要件は次である。

- 一意である
- 長期参照に耐える
- 人間の分類に依存しない
- 本文変更で変わらない
- 物理パス生成に使える
- 外部インデックスやグラフから参照できる

標準ID形式は ULID とする。

---

## 6.3 物理パスはIDから生成する

物理パスは、IDから生成したハッシュで分散する。

ULIDの先頭をフォルダに使ってはならない。

ULIDの先頭には時刻情報が含まれるため、大量インポート時に同じディレクトリへ偏る。

標準ルールは次である。

```
id = ULIDshard = sha256(id)[0:4]path = records/doc/{shard[0:2]}/{shard[2:4]}/{id}.md
```

例：

```
records/doc/ab/cd/01K2Z9P7Y8QWERTY1234567890.md
```

このパスは、本文が変わっても、タイトルが変わっても、分類が変わっても変えてはならない。

---

## 6.4 Front Matter は最小限に保つ

MarkdownのFront Matterは、正本として必要な最小限の情報を保持する。

大量のタグ、大量の関連リンク、大量のカテゴリ、大量のAI推論結果をFront Matterに詰め込んではならない。

Front Matterに入れるべきものは、安定性が高く、正本として保存する意味がある情報である。

例：

```
id: 01K2Z9P7Y8QWERTY1234567890type: web_articletitle: "Example Title"source_url: "https://example.com/article"created_at: "2026-06-07T00:00:00+09:00"captured_at: "2026-06-07T00:00:00+09:00"language: jacontent_hash: "sha256:..."schema_version: 1
```

候補概念や要約を入れる場合も、過剰に増やしてはならない。

---

## 6.5 本文には再利用可能な文脈を残す

本文には、あとから人間とAIが再利用できる文脈を残す。

単なるURLだけを保存してはならない。

リンク切れ、ページ改変、削除、ログイン制限、動的ページ化により、将来参照不能になる可能性があるためである。

保存すべき情報は、少なくとも次である。

- 何の情報か
- どこから取得したか
- いつ取得したか
- なぜ保存したか
- 主要な内容
- 再利用時の注意点
- 必要に応じた要約
- 必要に応じた引用または抜粋
- AIが文脈として使うための構造化された要点

---

## 7. AI利用の原則

## 7.1 AIは補助者であり、正本の唯一の判断者ではない

AIは知識整理を補助する。

AIは、次を生成できる。

- 要約
- 論点
- 候補概念
- 関連候補
- タイトル案
- 抽象化された教訓
- 再利用シーン
- 注意点
- 類似文書候補
- 重複候補
- 構造化メタデータ候補

しかし、AIの出力を無制限に正本へ流し込んではならない。

AIの出力は、ノイズを含む可能性がある。

AIが出した構造は、プログラムによる制約、正規化、検証、フィルタリングを通す。

---

## 7.2 AIタグ大量生成は禁止する

AIに大量のタグを生成させてはならない。

大量のタグは検索性を高めるのではなく、ノイズを増やす。

AIが出すべきものは、少数の候補概念である。

概念の統合、別名管理、停止概念、類似概念の結合、ビュー生成はプログラムで制御する。

---

## 7.3 AI要約は正本本文の代替ではない

AI要約は便利だが、元情報の代替ではない。

要約は情報を圧縮するが、同時に情報を失う。

したがって、可能な限り、出典・取得情報・本文相当の抽出・重要な抜粋・構造化要点を保持する。

AI要約だけを保存して、元情報の文脈を失ってはならない。

---

## 7.4 AIが使いやすい単位で保存する

このシステムは、人間だけでなくAIが使うことを前提にする。

そのため、各MarkdownはAIが文脈として読み込みやすい粒度・構造・メタデータを持つべきである。

望ましいMarkdownは、次を満たす。

- 1ドキュメントの主題が明確である
- 出典が明確である
- 保存理由が明確である
- 要点が抽出されている
- 必要な本文文脈が残っている
- 不要な装飾やノイズが少ない
- 後から検索・引用・再構成しやすい

---

## 8. Git運用の原則

## 8.1 Gitはバッチコミットする

1ページごとにコミットしてはならない。

コミット履歴が無駄に肥大化し、意味のある履歴確認が困難になる。

Gitへの保存は、一定件数、一定時間、一定処理単位、または明示的な同期タイミングでバッチコミットする。

---

## 8.2 Gitに巨大なRawデータを入れない

HTML、PDF、画像、動画、巨大JSON、クローラーキャッシュなどのRawデータは、原則としてGitに入れない。

必要な場合は、S3、R2、ローカルオブジェクトストレージ、Git LFSなどを使う。

Gitに保存するのは、長期的に差分管理しやすいMarkdown正本である。

---

## 8.3 Git差分を知識の変化として読める状態に保つ

Git差分は、人間が意味を読める状態でなければならない。

インデックス再生成、グラフ再生成、バイナリ更新、キャッシュ更新によって、Git差分を汚染してはならない。

Git履歴には、正本の追加・修正・削除・統合・訂正が記録されるべきである。

---

## 9. 検索と再利用の原則

## 9.1 検索はファイル一覧で行わない

知識検索は、Gitのファイル一覧やフォルダ巡回で行ってはならない。

検索は、SQLite FTS、DuckDB、ベクトルインデックス、概念インデックス、メタデータインデックスなどを使って行う。

ファイル一覧は保存構造であり、検索UIではない。

---

## 9.2 カテゴリは仮想ビューである

カテゴリは物理フォルダではない。

カテゴリは、メタデータ、本文、候補概念、出典、時系列、プロジェクト、利用目的などから生成される仮想ビューである。

同じ知識オブジェクトは、複数のビューに同時に現れてよい。

---

## 9.3 概念グラフは再生成可能である

概念グラフは、Markdown正本、概念定義、別名辞書、停止概念、AI候補、プログラムルールから生成される。

グラフは固定された正本ではない。

より良いルール、より良いモデル、より良い概念辞書が得られた場合、過去のMarkdown正本から再生成できるべきである。

---

## 10. スケール原則

## 10.1 100万件でも破綻しない設計にする

このシステムは、小規模メモ管理ではなく、大量知識蓄積を前提とする。

100件で便利な設計ではなく、100万件でも破綻しにくい設計を優先する。

そのため、次を禁止する。

- 人間向けフォルダ分類
- カテゴリを物理パスに入れること
- 本文ハッシュをファイル名にすること
- Markdownへの大量相互リンク直書き
- インデックスファイルの毎回Gitコミット
- AIによる大量タグ生成
- 1ページ1コミット
- Rawデータの無制限Git保存

---

## 10.2 ファイル配置は偏らせない

大量インポート時にも、特定ディレクトリにファイルが集中しないようにする。

ULIDの時刻部分をそのまま物理パスに使ってはならない。

IDをハッシュ化し、シャーディングして分散配置する。

---

## 10.3 更新に強い設計にする

本文が更新されても、パスは変えない。

タイトルが変わっても、パスは変えない。

カテゴリが変わっても、パスは変えない。

概念グラフが変わっても、Markdown本文を大量更新しない。

更新により大量の参照切れや差分汚染が発生する設計を避ける。

---

## 11. S3から継承する思想

このシステムは、S3そのものではない。

しかし、S3的な思想を継承する。

## 11.1 オブジェクトはIDで管理する

知識オブジェクトは、不変IDで管理する。

人間向けのフォルダ名やカテゴリ名に依存してはならない。

---

## 11.2 フォルダは実体ではない

カテゴリや階層は実体ではなくビューである。

物理配置と意味分類を分離する。

---

## 11.3 一覧で探さない

大量のオブジェクトをファイル一覧で探してはならない。

検索はインデックスで行う。

---

## 11.4 メタデータは検索可能にする

Front MatterとMarkdown本文からメタデータを抽出し、検索可能なインデックスへ変換する。

ただし、インデックスは正本ではない。

---

## 11.5 イベント駆動で処理する

知識は、次のようなパイプラインで処理される。

```
inbox  ↓ingest  ↓normalize  ↓analyze  ↓write markdown  ↓index  ↓graph build  ↓search / retrieve / export
```

処理途中のキャッシュや一時ファイルは、Git正本に混ぜない。

---

## 11.6 ライフサイクル管理を行う

Rawデータ、キャッシュ、インデックス、エクスポートは、保存コストと再生成可能性に応じて管理する。

古いRawデータや巨大ファイルは、必要に応じてGit外へ逃がす。

---

## 12. ディレクトリ思想

標準構造は次を基本とする。

```
knowledge-bucket/
  README.md

  config/
    kb.yml
    taxonomy.yml
    aliases.yml
    stop_concepts.yml

  prompts/
    analyzer_v1.md

  records/
    doc/
      ab/
        cd/
          01K2Z9P7Y8QWERTY1234567890.md
    concept/
      retrieval-augmented-generation.md
      graph-rag.md

  exports/
    README.md

  scripts/
    README.md

  .kb/
    index.sqlite
    vector.index
    cache/
    inbox/
```

`.kb/` はGit管理しない。

`.kb/` は作業領域であり、正本ではない。

---

## 13. `.gitignore` 原則

Gitに入れてはならないものを明示する。

標準例：

```
.kb/*.sqlite*.duckdb*.db*.lograw/cache/tmp/
```

この方針は、正本とキャッシュの分離を守るための最低条件である。

---

## 14. 不変条件

以下は、Git Knowledge Bucket の不変条件である。

## 14.1 すべての知識ドキュメントは不変IDを持つ

IDがないドキュメントを正本として扱ってはならない。

---

## 14.2 IDは本文更新で変えてはならない

本文の更新、タイトル変更、分類変更、要約変更によってIDを変えてはならない。

---

## 14.3 物理パスに意味分類を入れてはならない

`ai/rag/article.md` のような人間向け分類パスを正本配置に使ってはならない。

意味分類はビューで扱う。

---

## 14.4 正本はMarkdownである

正本はMarkdownで管理する。

SQLite、DuckDB、Parquet、Vector DB、Graph DBを正本にしてはならない。

---

## 14.5 インデックスは再生成可能でなければならない

インデックスが削除されても、Markdown正本から再生成できなければならない。

---

## 14.6 Markdownに大量の関連リンクを書いてはならない

関連構造はプログラムで生成する。

Markdownへの大量相互リンク直書きは禁止する。

---

## 14.7 AIに大量タグを生成させてはならない

AIは少数の候補概念を出す。

大量タグ付けはノイズ化するため禁止する。

---

## 14.8 Git差分をキャッシュ更新で汚してはならない

キャッシュ、インデックス、ログ、RawデータをGit差分に混ぜてはならない。

---

## 14.9 Gitはバッチコミットする

1ページごとのコミットを標準運用にしてはならない。

---

## 14.10 Rawデータは原則Git外に置く

巨大なRawデータはGit外で管理する。

Gitには、長期保存に適したMarkdown正本を置く。

---

## 15. 非ゴール

このシステムは、次を目的としない。

- 人間がフォルダを開いて整理するためのノート術
- Obsidian的な手動リンク網の構築
- AIに全知識分類を丸投げすること
- タグを増やして整理した気になること
- Gitを巨大バイナリ保管庫として使うこと
- 検索インデックスを正本として扱うこと
- フォルダ階層で世界の知識体系を表現すること
- 保存しただけで活用した気になること
- UI都合で正本構造を歪めること
- 短期的な見やすさのために長期的な再利用性を犠牲にすること

---

## 16. 成功条件

Git Knowledge Bucket が成功している状態は、次である。

```
大量の情報が保存されているかつ各情報に不変IDがあるかつMarkdown正本として読めるかつGit履歴で変更を追えるかつ検索インデックスを再生成できるかつ概念グラフを再生成できるかつAIが必要な文脈を取り出せるかつ人間が正本へ戻って検証できる
```

逆に、情報が大量に保存されていても、次の状態なら失敗である。

```
フォルダ分類が破綻しているIDが安定していないパス変更で参照が切れるMarkdownがタグやリンクで汚染されているインデックスが正本化しているGit差分がキャッシュ更新で読めないAI生成タグがノイズ化している検索しても再利用できる文脈が取れない
```

---

## 17. 最上位KPI

最上位KPIは、保存件数ではない。

最上位KPIは、次である。

```
保存された知識が、将来の人間またはAIによって再利用可能な文脈として取り出せる割合
```

補助KPIは次である。

- 正本Markdown件数
- ID付きドキュメント率
- 出典付きドキュメント率
- 再インデックス成功率
- 重複検出率
- 検索ヒット後の有用率
- AI文脈投入時の再利用成功率
- 概念グラフ再生成成功率
- Git差分の可読性
- キャッシュ混入率の低さ
- RawデータのGit混入率の低さ

---

## 18. 設計判断の優先順位

迷った場合は、次の優先順位で判断する。

```
1. 正本の長期保存性2. IDと参照の安定性3. 再生成可能性4. Git差分の可読性5. AIによる再利用性6. 人間による検証可能性7. 大量データへの耐性8. 実装の単純さ9. UI上の見やすさ
```

UI上の見やすさを理由に、正本構造を壊してはならない。

短期的な便利さを理由に、長期的な再利用性を犠牲にしてはならない。

---

## 19. 最終定義

Git Knowledge Bucket の究極のゴールは、情報をフォルダに整理することではない。

Git Knowledge Bucket の究極のゴールは、あらゆる知識情報を、不変IDを持つMarkdown正本としてGitに保存し、検索・分類・関連グラフ・ベクトル索引などの派生構造をすべて再生成可能なキャッシュとして扱うことで、人間とAIが長期にわたり検証・検索・接続・再利用できる知識基盤を構築することである。

このシステムにおいて、Gitは見た目の整理棚ではなく、知識正本の台帳である。

Markdownは正本である。

IDは不変である。

パスは分類ではない。

カテゴリはビューである。

インデックスはキャッシュである。

グラフは再生成物である。

AIは候補生成者であり、構造の唯一の支配者ではない。

この原則を守る限り、実装は変えてよい。

この原則を壊す実装は、どれほど便利に見えても採用してはならない。
